## 1. 基本操作のおさらい

### 起動方法

```bash
# 通常起動（対話モード）
claude

# 初期プロンプト付きで起動
claude "VPCのTerraformコードを作成して"

# 前回のセッションを続行
claude -c

# 前回のセッションを再開（セッション選択画面を表示）
claude -r
```

### セッションの概念

Claude Code は**セッション**単位で会話を管理します。セッションには以下の特徴があります：

- 1つのセッション内では、前の会話の内容を覚えている（コンテキストが維持される）
- `claude` で起動するたびに新しいセッションが始まる
- `claude -c` で直前のセッションを続行できる
- `claude -r` で過去のセッション一覧から選んで再開できる

> **Tip**: 同じタスクの続きを行うときは `claude -c` を使うと、これまでの作業内容を引き継げます。

## 2. スラッシュコマンド一覧

Claude Code の対話中に使えるスラッシュコマンドです。

| コマンド | 説明 |
|---------|------|
| `/help` | ヘルプを表示 |
| `/compact` | 会話のコンテキストを要約して圧縮する（トークン節約） |
| `/clear` | 会話履歴をクリアして新しいセッションを開始 |
| `/cost` | 現在のセッションのトークン使用量・コストを表示 |
| `/model` | 使用中のモデルを表示・変更 |
| `/memory` | CLAUDE.md（プロジェクト記憶）を編集 |
| `/permissions` | 権限設定を表示・変更 |
| `/config` | 設定を表示 |

### よく使うコマンド

```
# コンテキストが長くなってきたら圧縮
/compact

# トークン使用量の確認
/cost

# 権限の確認
/permissions
```

## 3. キーボードショートカット

| ショートカット | 説明 |
|--------------|------|
| `Enter` | プロンプトを送信 |
| `Shift + Enter` | 改行（複数行入力） |
| `Escape` | 実行中の処理をキャンセル（中断） |
| `Ctrl + C` | Claude Code を終了 |
| `Tab` | ファイルパスの補完 |

> **重要**: `Escape` は処理の一時中断、`Ctrl + C` は Claude Code 自体の終了です。途中で方針を変えたいときは `Escape` を使いましょう。

## 4. モードの解説

Claude Code には、用途に応じて切り替えられる複数のモードがあります。

### モード一覧

| モード | 切り替え方法 | 説明 |
|-------|------------|------|
| **通常モード** | デフォルト | ファイル編集やコマンド実行を伴う通常の対話モード |
| **Plan モード** | `Shift + Tab` または `/plan` | 実行せずに計画・方針だけを立てるモード |

---

### 通常モード（デフォルト）

起動直後のモードです。プロンプトに応じてファイルを作成・編集したり、コマンドを実行したりします。

```
# 通常モードでの操作例
VPCのTerraformコードを作成してください
→ コードを生成してファイルに書き込む
```

---

### Plan モード

**実行を伴わず、作業計画だけを立てるモード**です。Claude Code が何をしようとしているかを事前に確認したいときに使います。

#### 切り替え方法

- `Shift + Tab` を押す（トグル切り替え）
- 対話中に `/plan` と入力する

#### Plan モードでできること

- コードを書く前に「どういう構成にするか」を相談できる
- 複数の選択肢を提示してもらい、方針を決めてから実装に進める
- 誤った方向で大量のコードが生成されるリスクを減らせる

#### Plan を確認する方法

Plan モードで Claude が立てた計画は、**チャット画面に直接テキストとして出力されます**。ファイルは一切変更されません。

出力された計画を見て、以下を確認しましょう：

- 作成されるファイル・リソースの一覧が意図通りか
- ファイルの構成（`main.tf` / `variables.tf` など）が適切か
- 命名規則（prefix など）が正しく反映されているか
- 不要なリソースが含まれていないか

計画に問題があれば、通常モードに戻らずそのまま Plan モード内で修正を依頼できます。

```
# 計画の修正例（Plan モード内で）
サブネットは1つではなくパブリックとプライベートの2つにしてください
```

#### 通常モードとの使い分け

| 状況 | 推奨モード |
|-----|-----------|
| 初めて取り組むリソースを作成する | **Plan モード**で構成を確認してから通常モードへ |
| 既存リソースへの小さな変更（タグ追加など） | **通常モード**で直接実施 |
| `terraform destroy` など取り返しのつかない操作 | **Plan モード**で影響範囲を確認してから通常モードへ |
| エラーの原因調査 | **通常モード**（調査・説明だけなら実行リスクは低い） |
| 複数ファイルにまたがる大きな変更 | **Plan モード**で全体像を把握してから通常モードへ |

#### 使用例

```
# Step 1: Plan モードで計画を確認
[Shift+Tab で Plan モードに切り替え]

VPCとEC2を作成するTerraformコードの構成を考えてください

→ Claude が実際にファイルを作らず、構成案だけをテキストで提示する
→ 内容を確認し、必要なら修正を依頼する

# Step 2: 通常モードに戻って実装
[Shift+Tab で通常モードに戻す]

その計画で実装してください
```

> **Tip**: 複雑なタスクや不慣れな作業は、まず Plan モードで構成を確認してから実装に進むと安全です。

## 5. プロンプトの書き方

### 基本原則：具体的に、明確に

Claude Code は指示が具体的であるほど、正確な結果を返します。

#### 悪い例と良い例

| 悪い例 | 良い例 |
|-------|-------|
| VPCを作って | CIDRが10.0.0.0/16のVPCをTerraformで作成してください。名前は`${var.prefix}-vpc`にしてください |
| EC2を修正して | terraform/vpc-ec2/main.tfのEC2インスタンスにタグ`Environment = "dev"`を追加してください |
| エラーを直して | `terraform plan`を実行して、エラーが出たら原因を説明して修正してください |

### テクニック 1：段階的に指示する

一度に大きな指示を出すよりも、段階的に進めるのが効果的です。

```
# Step 1
VPCとサブネットのTerraformコードを作成してください

# Step 2（生成されたコードを確認してから）
terraform initとplanを実行してください

# Step 3（planの結果を確認してから）
問題なさそうなのでapplyしてください
```

### テクニック 2：コンテキストを与える

何をしたいのか、背景も伝えると精度が上がります。

```
# 背景情報付きの指示
Session 1で作成したVPC（terraform/vpc-ec2/）にパブリックサブネットを追加したいです。
既存のVPCのCIDRは10.0.0.0/16です。
サブネットのCIDRは10.0.1.0/24にしてください。
```

### テクニック 3：出力形式を指定する

```
# 説明を求める場合
このTerraformコードの各リソースが何をしているか、表形式で説明してください

# ファイル構成を指定する場合
以下のファイル構成でTerraformコードを作成してください：
- main.tf: プロバイダー設定とVPC
- variables.tf: 変数定義
- outputs.tf: 出力定義
```

### テクニック 4：確認してから実行を依頼する

インフラに変更を加えるコマンドは、確認ステップを挟むと安全です。

```
terraform planの結果を見せてください。問題なければapplyします。
```

### テクニック 5：エラー時は自分の理解を添えて質問する

エラーが出たとき、「直して」だけだと Claude Code が直してくれても自分の力にはなりません。エラーメッセージを読んで、自分なりの解釈を添えて質問しましょう。

```
terraform planで以下のエラーが出ました：
  Error: Reference to undeclared resource

リソース名のtypoだと思いますが、合っていますか？
```

仮説が間違っていても問題ありません。Claude Code が正しい原因を説明してくれます。

## 6. CLAUDE.md によるプロジェクト記憶

### CLAUDE.md とは

プロジェクトルートに配置する `CLAUDE.md` は、Claude Code に対する「永続的な指示書」です。セッションを開始するたびに自動で読み込まれます。

このハンズオンでは、以下のような情報が `CLAUDE.md` に記載されています：

- リソース命名規則（prefix の使用）
- SSH 鍵のパス
- プロジェクトの構成

### CLAUDE.md の編集

```
# 対話中にスラッシュコマンドで編集
/memory
```

### 活用例

ハンズオンの途中で「毎回同じことを伝えるのが面倒」と感じたら、それは CLAUDE.md に書くべき情報です。

```markdown
## 追加メモ
- EC2のIPアドレス: 54.xxx.xxx.xxx
- 現在のセッション: Session 4（Ansible）
- Ansibleのインベントリファイル: ansible/inventory.ini
```

> **Tip**: `/memory` コマンドで追記すれば、次回以降のセッションでも自動的に参照されます。

## 7. 権限モデルの理解

### 承認が必要な操作

Claude Code は安全のため、以下の操作を実行する前に承認を求めます：

| 操作 | 例 |
|-----|---|
| ファイルの作成・編集 | Terraform ファイルの作成、既存コードの変更 |
| ターミナルコマンドの実行 | `terraform apply`、`ssh`、`ansible-playbook` |
| ファイルの削除 | 不要なファイルの削除 |

### 承認の選択肢

コマンド実行の承認時には、以下の選択肢があります：

| 選択肢 | 意味 |
|-------|------|
| **Allow once (y)** | 今回だけ許可 |
| **Allow always** | このコマンド/ツールを今後も常に許可 |
| **Deny (n)** | 拒否（実行しない） |

### 権限設定のカスタマイズ

頻繁に使うコマンドは `Allow always` にすると、毎回の承認が不要になります。

```
# 権限設定の確認
/permissions
```

> **注意**: `terraform apply` や `rm` など破壊的な操作は、毎回確認する（Allow once）のが安全です。

## 8. IaC ワークフローでの活用パターン

### パターン 1：Terraform コードの生成 → 実行

```
# 1. コード生成を依頼
10.0.0.0/16のVPCを作成するTerraformコードを書いてください。
ファイルはterraform/vpc-ec2/main.tfに作成してください。

# 2. 確認と実行
terraform initとplanを実行してください

# 3. 適用
planの結果に問題がないのでapplyしてください
```

### パターン 2：既存コードの修正

```
# 差分を見ながら修正
terraform/vpc-ec2/main.tfのEC2インスタンスに以下の変更を加えてください：
- インスタンスタイプをt2.microからt2.smallに変更
- タグにEnvironment = "dev"を追加
変更後にplanを実行して差分を確認させてください
```

### パターン 3：エラーの調査と修正

```
# terraform applyでエラーが出た場合
terraform applyを実行してください。エラーが出たら原因を分析して修正案を提示してください。
```

### パターン 4：Ansible Playbook の作成と実行

```
# Playbook 作成
対象ホスト54.xxx.xxx.xxxにnginxをインストールするAnsible Playbookを作成してください。
SSH鍵は~/workspace/keys/training-keyを使います。

# 実行
ansible-playbookを実行してください
```

### パターン 5：調査・確認作業

```
# AWS リソースの確認
AWS CLIを使って、現在のVPC一覧を表示してください

# ファイルの内容確認
terraform/vpc-ec2/main.tfの内容を説明してください

# 接続確認
54.xxx.xxx.xxxにSSHで接続して、nginxが動いているか確認してください
```

## 9. トラブル時の対処法

### コンテキストが長くなって動作が遅い

```
/compact
```

コンテキストを要約・圧縮して、レスポンスを改善します。それでも改善しない場合は：

```
/clear
```

で新しいセッションを開始してください。

### 意図しない方向に進んでしまった

`Escape` キーで処理を中断し、方針を修正する指示を出しましょう。

```
# Escape で中断した後
いえ、そうではなく〜してください
```

### Claude Code が同じミスを繰り返す

具体的な制約条件を明示しましょう。

```
# 制約条件を明示
以下の条件を守ってください：
- リソース名には必ず${var.prefix}をつけること
- CIDRは10.0.0.0/16を使うこと
- AZはap-northeast-1aを使うこと
```

### セッションが途切れた

```bash
# 直前のセッションを続行
claude -c

# 過去のセッション一覧から選択
claude -r
```

## 10. よくある落とし穴と対策

### 落とし穴 1：prefix を付け忘れる

**問題**: リソース名に prefix を付けないと、他の受講者のリソースと衝突します。

**対策**: 最初に CLAUDE.md の命名規則を確認してもらうか、指示に必ず含める。

```
リソース名には必ず${var.prefix}を含めてください
```

### 落とし穴 2：一度に大きな指示を出しすぎる

**問題**: 「VPC、サブネット、セキュリティグループ、EC2を全部作って」のような大きな指示は、ミスが入り込みやすい。

**対策**: リソースごとに段階的に作成し、都度 `terraform plan` で確認する。

### 落とし穴 3：plan を確認せずに apply する

**問題**: 意図しないリソースの削除や変更が含まれている可能性がある。

**対策**: 必ず `terraform plan` の結果を確認してから `apply` を指示する。

```
planを実行して結果を見せてください。確認してからapplyを指示します。
```

### 落とし穴 4：エラーメッセージを読まずに再実行する

**問題**: エラーの内容を理解せずに再実行しても、同じエラーが繰り返される。AI に丸投げし続けると、何も学べないまま時間だけが過ぎる。

**対策**: エラーが出たら、まず**自分でエラーメッセージを読む**。その上で Claude Code に相談する。

```
以下のエラーが出ています：
  Error: creating Security Group: InvalidGroup.Duplicate

「同じ名前のセキュリティグループが既に存在する」という意味だと思います。
既存のリソースを確認して、重複しない名前に変更する方法を教えてください。
```

> **ポイント**: 「自分はこう読んだけど合っていますか？」という聞き方をすると、エラーの読み方そのものがスキルとして身に付きます。

### 落とし穴 5：コンテキストの断絶

**問題**: 新しいセッションを開始すると、前のセッションの作業内容を覚えていない。

**対策**:
- 同じタスクの続きは `claude -c` で前回セッションを続行する
- 新しいセッションで始める場合は、現状を簡潔に伝える

```
Session 1で、VPCとサブネットまで作成済みです。
terraform/vpc-ec2/main.tfに既存のコードがあります。
次にインターネットゲートウェイを追加したいです。
```

---

## まとめ：効果的に使うための3つの心がけ

1. **具体的に指示する** — 曖昧な指示は曖昧な結果を生む。ファイルパス、CIDR、リソース名など具体的に伝える
2. **段階的に進める** — 大きなタスクは小さなステップに分けて、各ステップで結果を確認する
3. **確認してから実行する** — 特にインフラ変更（`apply`、`destroy`）は plan 結果を確認してから
