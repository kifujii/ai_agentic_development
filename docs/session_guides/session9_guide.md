# セッション9：インシデント対応とポストモーテム（1.5時間）

## シナリオ

金曜日の夕方 17:30。退勤しようとしたところに Slack の通知が鳴りました。

> **[ALERT] CPU使用率が閾値を超えました**
> **[ALERT] ヘルスチェックが失敗しています**
> **[ALERT] ディスク使用率 95%**

複数のアラートが同時に発報しています。
チームメンバーはすでに帰宅しており、あなたと AI だけが頼りです。

---

## 🎯 ゴール

以下の状態になっていれば完了です：

1. **障害が復旧している**
   - サービスが正常にアクセスできる状態に戻っている
   - アラートが解消されている

2. **ポストモーテム（障害報告書）が作成されている**
   - タイムライン（いつ何が起きたか）
   - 根本原因の特定
   - 影響範囲
   - 復旧手順の記録
   - 再発防止策

3. **再発防止策が実装されている**
   - ポストモーテムで挙げた対策のうち、少なくとも1つを Terraform / Ansible で実装

---

## 進め方

### Phase 1: 状況把握と復旧（30分）

障害が発生しています。まずは**復旧を最優先**にしてください。

- AI と一緒に、何が起きているか調査する
- 原因の特定よりも、まずサービスの復旧を優先する
- 復旧作業の各ステップを記録に残す（後でポストモーテムに使う）

> **注意**: 障害の原因は1つとは限りません。1つ直して安心しないでください。

### Phase 2: 根本原因の分析（20分）

復旧した後、改めて原因を分析してください。

- 表面的な症状ではなく、**なぜそれが起きたのか**を掘り下げる
- AI に「5 Whys（なぜなぜ分析）」を手伝ってもらうのも有効

### Phase 3: ポストモーテム作成（20分）

`docs/` 配下にポストモーテムドキュメントを作成してください。

形式は自由ですが、最低限以下を含めてください：
- 発生日時と検知方法
- タイムライン
- 根本原因
- 影響範囲
- 復旧手順
- 再発防止策（短期・中長期）

### Phase 4: 再発防止策の実装（20分）

ポストモーテムで挙げた再発防止策のうち、時間内に実装できるものを1つ以上実装してください。

---

## 障害の準備（講師向け）

このセッションを開始する前に、講師が EC2 インスタンスに対して障害を仕込む必要があります。

<details>
<summary>🔧 障害の仕込み方（講師が実施）</summary>

受講者の EC2 に SSH して、以下のような操作を行ってください（組み合わせ可）：

```bash
# nginx を停止
sudo systemctl stop nginx

# ディスクを圧迫する大きなファイルを作成
sudo fallocate -l 5G /tmp/bloat.img

# CPU を使い切るプロセスを起動
nohup yes > /dev/null &
```

どの障害を仕込んだかは受講者に伝えないでください。

</details>

</details>
