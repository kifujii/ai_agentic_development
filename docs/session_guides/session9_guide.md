# セッション9：インシデント対応とポストモーテム（1時間）

## 学習目標

- インフラレベルとサーバーレベルの複合障害を AI と協力して調査・復旧する
- Terraform によるインフラドリフト検知と復旧を体験する
- 障害報告書（ポストモーテム）の作成と再発防止策の提案を行う

## シナリオ

月曜朝、出社するとチームの Slack チャンネルが騒がしい。

> 「Web サイトにアクセスできない」
> 「SSH も繋がらない」
> 「サーバーが応答しない」

週末当番だった先輩のメモにはこう書かれている：

> 「セキュリティの問題があったので対応した。サーバーの設定も少しいじった。詳細は月曜に共有する」

しかし先輩は体調不良で休み。何を変えたのかは分かりません。
あなたと AI だけで状況を把握し、復旧するしかありません。

> 💡 **セッション4との違い**: セッション4ではサーバー内部の障害を Ansible で対応しました。このセッションでは **AWS インフラの設定変更とサーバー内部の障害が同時に起きた複合障害** を扱います。`terraform plan` によるドリフト検知とポストモーテム作成まで行います。

---

## 🎯 ゴール

以下の状態になっていれば完了です：

1. **Web サイトが復旧している**
   - ブラウザで `http://<EC2のIP>` にアクセスして nginx のページが表示される

2. **すべての問題が解消されている**
   - AWS インフラの設定が正しい状態に戻っている
   - サーバー内部の問題もすべて修復されている

3. **ポストモーテム（障害報告書）が作成されている**
   - 何が起きていたか（発見した問題の一覧）
   - どう復旧したか（手順の記録）
   - なぜ起きたか（根本原因）
   - 再発防止策

---

## 進め方

> この障害は **複数のレイヤー** にまたがっています。1つ直して終わりではありません。

### 事前準備：障害を仕込む

セッション1-2 で作成した `terraform/vpc-ec2` 環境（VPC + EC2 + nginx 構成）を対象に、障害を仕込みます。

```bash
./scripts/break_session9.sh
```

スクリプトが完了したら、その内容は **忘れて** ください。ここからは「先輩が何かしたらしいが、何を変えたか分からない」状態として進めます。

> ⚠️ スクリプトを実行するとターミナルに何が壊れたか一部表示されますが、サーバー内部の2件はランダムのため、すべての障害を把握しているわけではありません。

---

### Phase 1: 状況把握と復旧（35分）

まずは現状を確認してください。何が起きているのか分からない状態からスタートです。

AI と一緒に調査を進めてください。調査の過程で見つけた問題は、都度メモしておくと後でポストモーテムに使えます。

<details>
<summary>💡 どこから手をつければいいか分からないとき</summary>

- まず AWS の状態を確認しましょう。EC2 は動いていますか？
- `terraform plan` を実行すると、Terraform が管理している「あるべき姿」と現状の差分が分かります
- インフラが直っても、サーバーの中にも問題が残っているかもしれません

</details>

<details>
<summary>💡 EC2 の IP アドレスが変わった場合</summary>

EC2 を停止・起動すると、パブリック IP アドレスが変わります。
`terraform output` や `aws ec2 describe-instances` で新しい IP を確認してください。

</details>

### Phase 2: ポストモーテム作成（20分）

復旧が完了したら、AI と一緒に `docs/` 配下にポストモーテムドキュメントを作成してください。

以下の内容を含めてください：

- **タイムライン**: いつ何を発見し、どう対処したか
- **発見した問題一覧**: インフラ・サーバーそれぞれの問題
- **根本原因**: なぜこのような状態になったのか
- **復旧手順**: どのコマンド・ツールで復旧したか
- **再発防止策**: 同じことが起きないようにするにはどうすべきか

> 💡 AI に「SRE のポストモーテムテンプレートを教えて」と聞くと、業界標準のフォーマットを提案してくれます。

### Phase 3: 振り返り（5分）

| 観点 | 確認 |
|------|------|
| `terraform plan` でインフラの差分を検出できた | ☐ |
| インフラ復旧後にサーバー内部の問題にも気づけた | ☐ |
| ポストモーテムに全ての問題が記録されている | ☐ |
| 再発防止策を具体的に提案できた | ☐ |

## ヒント

<details>
<summary>💡 調査のアプローチ</summary>

この障害は **AWS インフラ** と **サーバー内部** の両方にまたがっています。

**インフラの調査:**
- `terraform plan` で Terraform 管理下のリソースの差分を確認
- `aws ec2 describe-instances` でインスタンスの状態を確認
- `aws ec2 describe-security-groups` でセキュリティグループのルールを確認

**サーバーの調査（SSH 接続後）:**
- `systemctl` でサービスの状態を確認
- `/var/log/` 配下のログを確認
- `df -h` でディスク使用量を確認
- 設定ファイルの内容を確認

</details>

<details>
<summary>💡 復旧の順序</summary>

すべてを一度に直す必要はありません。順番に考えましょう：

1. まず **EC2 にアクセスできる状態** を目指す
2. アクセスできたら **サーバー内部の問題** を調査・修復
3. 最後に **Web サイトが表示される** ことを確認

</details>

<details>
<summary>💡 Terraform でのインフラ復旧</summary>

Terraform は「あるべき姿」をコードで定義しています。`terraform plan` で差分を確認し、`terraform apply` で復旧できます。

ただし、Terraform が管理していない変更（例: インスタンスの停止/起動）は `terraform plan` に表示されない場合があります。

</details>

## 📚 事前準備

> ⚠️ **環境変数が未設定の場合**: `echo $TF_VAR_prefix` で値が表示されない場合は講師に確認してください。

- セッション1-2 で構築した VPC + EC2 環境（`terraform/vpc-ec2`）の Terraform state が存在すること
- Ansible 環境が構築済みであること（セッション4を完了済みであること）
- EC2 が起動中であること（`terraform -chdir=terraform/vpc-ec2 output instance_public_ip` で IP が取れること）
