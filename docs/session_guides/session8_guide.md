# セッション8：本番リリースの設計判断（1.5時間）

## 学習目標

- やや難易度の高いインフラ要求を満たす構成をAI Agentと導出する
- 技術選択の根拠を説明するドキュメントを作成する

## シナリオ

セッション7で作ったデモ API が好評で、上司からこう言われました。

> 「さっきのデモ API、評判が良かったから**本番環境としてリリース**することになった。
> ただし、今のままじゃ足りない。最低限 **データの永続化** と **エラー監視** は必要だ。
> あと、月額コストを見積もってほしい。**月3万円以内** で収まるか確認してくれ。
> 来週の会議で構成案とコスト見積もりを説明してくれ。」

現在の構成は Lambda + API Gateway だけで、データは保存されず、エラーが起きても気づけません。

---

## 🎯 ゴール

以下の状態になっていれば完了です：

1. **データの永続化が実装されている**
   - API がデータを保存・取得できる（DynamoDB など）
   - API を叩くたびに結果がちゃんと蓄積される

2. **監視・アラートが設定されている**
   - Lambda のエラーや API Gateway の異常を検知できる仕組みがある

3. **設計判断の根拠ドキュメントが作成されている**
   - なぜその構成を選んだのか
   - コストの見積もり（月額3万円以内に収まる根拠）
   - トレードオフ（何を妥協したか）

---

## 進め方

### Phase 1: 設計と見積もり（30分）

**Plan モード**（対話中に `Shift + Tab` または `/plan` で切り替え）で、以下を AI と議論してください：

- デモ API にデータの永続化を追加するには何が必要か
- サーバーレス構成でのデータストアの選択肢は何か
- エラー監視にはどんな方法があるか
- それぞれの月額コストはいくらか
- 月3万円の制約の中でどこまでできるか

正解は1つではありません。**自分がチームに説明できる構成**を選んでください。

### Phase 2: 構築（35分）

設計に基づいて、セッション7で作った Terraform コードを拡張してください。

新しいリソースを追加する形でも、ディレクトリを分けて作り直しても構いません。

### Phase 3: 検証とドキュメント作成（25分）

- 動作テスト: `curl` でデータの保存・取得が正しく動くことを確認
- 監視テスト: 意図的にエラーを発生させてアラームが反応するか確認
- 設計判断ドキュメント: AI と一緒に `docs/` 配下にドキュメントを作成

---

## ヒント

<details>
<summary>💡 データ永続化のキーワード</summary>

- DynamoDB（サーバーレスと相性が良い NoSQL データベース）
- Lambda から DynamoDB にアクセスするには IAM 権限が必要
- API のエンドポイントを増やす（POST で保存、GET で取得など）

</details>

<details>
<summary>💡 監視のキーワード</summary>

- CloudWatch Alarms（Lambda エラー率、API Gateway 5xx レート）
- CloudWatch Logs（Lambda の実行ログは自動で出力される）
- Lambda の DLQ（Dead Letter Queue）でエラー時の情報を保持

</details>

<details>
<summary>💡 コスト見積もりのアプローチ</summary>

- Lambda: リクエスト数 × 実行時間で課金（月100万リクエストまで無料枠あり）
- API Gateway: リクエスト数で課金
- DynamoDB: 読み書きキャパシティで課金（オンデマンドモードが小規模向き）
- AI に「この構成で月額いくらか見積もって」と聞けます

</details>

<details>
<summary>💡 設計判断ドキュメントに書くこと</summary>

- 選んだ構成の概要図
- 代替案との比較（なぜ他を選ばなかったか）
- コストの内訳
- 残っているリスクと対策

</details>

## 📚 事前準備

> ⚠️ **環境変数が未設定の場合**: `echo $TF_VAR_prefix` で値が表示されない場合は講師に確認してください。

- セッション7で構築した Lambda + API Gateway の環境があること
- セッション7のリソースを削除してしまった場合は、再度 `terraform apply` で構築してから開始してください

---

## ⚠️ リソースの削除

このセッションで追加・変更したリソースは、完了後に削除してください：

```bash
terraform -chdir=terraform/<作成したディレクトリ> destroy
```

> 💡 DynamoDB テーブルやCloudWatch Alarm も `terraform destroy` でまとめて削除されます。手動で作成したリソースがある場合は、別途 AWS CLI で削除してください。
