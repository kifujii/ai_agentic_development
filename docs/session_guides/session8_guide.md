# セッション8：本番リリースの設計判断（1.5時間）

## 学習目標

- やや難易度の高いインフラ要求を満たす構成を AI Agent と導出する
- 技術選択の根拠を説明するドキュメントを作成する

## シナリオ

セッション7で作ったメッセージボード API がハッカソンで好評でした。上司からこう言われました。

> 「あのメッセージボード、**チーム全体で日常的に使いたい** という声が出てる。
> 本番運用するなら最低限こうしてほしい：
>
> 1. **エラーが起きたらすぐ気づける** ようにしてくれ
> 2. **月額コストを見積もって** 上に説明できるようにしてくれ
> 3. 誰でも使えちゃうのはまずいから、**API キー** か何か入れてくれ
>
> 来週の会議で構成案とコスト見積もりを説明してくれ。」

現在の構成はセッション7で作った MVP です。動くけれど、監視もセキュリティも入っていません。

---

## 🎯 ゴール

以下の状態になっていれば完了です：

1. **監視・アラートが設定されている**
   - Lambda のエラーや API Gateway の異常を検知できる仕組みがある
   - 意図的にエラーを起こして、アラームが反応することを確認済み

2. **アクセス制御が追加されている**
   - API キー、使用量プラン、またはその他の認証手段が1つ以上設定されている
   - API キーなしのリクエストが拒否されることを確認済み

3. **設計判断の根拠ドキュメントが作成されている**
   - なぜその構成を選んだのか
   - コストの見積もり（月額3万円以内に収まる根拠）
   - トレードオフ（何を妥協したか）

---

## 進め方

### Phase 1: 設計と見積もり（30分）

**Plan モード**（対話中に `Shift + Tab` または `/plan` で切り替え）で、以下を AI と議論してください：

- 今の構成に監視を追加するには何が必要か
- API のアクセス制御にはどんな方法があるか（API キー、IAM 認証、Cognito など）
- それぞれの月額コストはいくらか
- 月3万円の制約の中でどこまでできるか

正解は1つではありません。**自分がチームに説明できる構成**を選んでください。

### Phase 2: 構築（35分）

設計に基づいて、セッション7で作った Terraform コードを拡張してください。

新しいリソースを追加する形でも、ディレクトリを分けて作り直しても構いません。

### Phase 3: 検証とドキュメント作成（25分）

- **監視テスト**: 意図的にエラーを発生させて（存在しないエンドポイントを叩く等）、アラームが反応するか確認
- **アクセス制御テスト**: API キーなしでリクエストして 403 が返ることを確認。API キー付きで正常にアクセスできることを確認
- **設計判断ドキュメント**: AI と一緒に `docs/` 配下にドキュメントを作成

## ヒント

<details>
<summary>💡 監視のキーワード</summary>

- CloudWatch Alarms（Lambda エラー率、API Gateway 5xx レート）
- CloudWatch Logs（Lambda の実行ログは自動で出力される）
- セッション5で学んだ CloudWatch の知識がここでも活きます

</details>

<details>
<summary>💡 アクセス制御のキーワード</summary>

- API Gateway の API キー + 使用量プラン（最もシンプル）
- `curl` に `-H "x-api-key: <キー>"` を付けてアクセス
- Terraform で `aws_api_gateway_api_key` や `aws_api_gateway_usage_plan` を設定

</details>

<details>
<summary>💡 コスト見積もりのアプローチ</summary>

- Lambda: リクエスト数 × 実行時間で課金（月100万リクエストまで無料枠あり）
- API Gateway: リクエスト数で課金
- DynamoDB: 読み書きキャパシティで課金（オンデマンドモードが小規模向き）
- CloudWatch Alarms: アラームあたり月額 $0.10
- AI に「この構成で月額いくらか見積もって」と聞けます

</details>

<details>
<summary>💡 設計判断ドキュメントに書くこと</summary>

- セッション7からの構成変更の概要図
- 監視の仕組み（何を監視し、どんな条件でアラートするか）
- アクセス制御の方式と選定理由
- コストの内訳
- 残っているリスクと対策

</details>
