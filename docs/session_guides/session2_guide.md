# セッション2：Webアプリケーションを公開しよう（必須・2時間）

## 🎯 このセッションのゴール

セッション1で構築したEC2にWebサーバーをインストールし、ブラウザからアクセスできるWebアプリケーションを公開します。**すべての作業をAgentに指示して実行**します。

![目標構成](../images/session2_target.svg)

### このセッションで変更・追加するリソース

| リソース | 変更内容 |
|---------|---------|
| セキュリティグループ | HTTP(80) のインバウンドルールを追加 |
| EC2 上のソフトウェア | nginx（Webサーバー）をインストール |
| Webコンテンツ | カスタムHTMLページをデプロイ |

> 🎓 このセッションでは「Terraform でインフラを変更 → Agentがサーバーにソフトウェアを導入 → Agentがアプリをデプロイ」という **一連の流れ** をすべてAgent経由で体験します。

---

## 📚 事前準備

- セッション1が完了していること（VPC/EC2が構築済み）
- EC2のIPアドレスを確認してメモ：

```bash
cd terraform/vpc-ec2
terraform output instance_public_ip
cd ../..  # プロジェクトルートに戻る
```

> ⚠️ **作業ディレクトリについて**: Continueへのプロンプトは **プロジェクトルート** から実行してください。

## Step 1: セキュリティグループに HTTP を追加しよう（25分）

### やること

現在のEC2セキュリティグループはSSH(22)のみ許可していますが、Webアプリを公開するためにHTTP(80)のアクセスも許可する必要があります。

### ゴール

`terraform/vpc-ec2/` の既存コードを修正して apply する：

- 既存のセキュリティグループに **HTTP(80)** のインバウンドルールを追加
- ソース: `0.0.0.0/0`（全体に公開）

> 💡 **ヒント**: `aws_security_group` リソースの `ingress` ブロックを追加します。SSH のルールはそのまま残し、HTTP 用のルールを新たに追加してください。

<details>
<summary>📝 プロンプト例</summary>

```
terraform/vpc-ec2/ の既存コードで、EC2のセキュリティグループに HTTP(80) のインバウンドルールを追加してください。
ソースは 0.0.0.0/0 で、既存の SSH ルールはそのまま残してください。

terraform apply まで実行してください。
```

</details>

### 確認

まず `terraform output` で全出力を確認します：

```bash
cd terraform/vpc-ec2
terraform output
```

AWSコンソールまたは以下のコマンドで、HTTP(80)ルールが追加されていることを確認：

```bash
# セッション1で security_group_id を output に定義している場合
aws ec2 describe-security-groups \
  --group-ids "$(terraform output -raw security_group_id)" \
  --query 'SecurityGroups[0].IpPermissions'
cd ../..
```

> 💡 `terraform output -raw security_group_id` でエラーが出る場合は、`terraform output` の結果からSG IDを確認し、直接指定してください：
> ```bash
> aws ec2 describe-security-groups --group-ids sg-xxxxx --query 'SecurityGroups[0].IpPermissions'
> ```

HTTP(80) のルールが表示されれば OK ✅

## Step 2: Agent に nginx のインストールを指示しよう（20分）

### やること

Agentに「EC2にSSHで接続してnginxをインストール・起動して」と指示します。受講者が手動でSSHやコマンドを実行する必要はありません。

### ゴール

AgentがEC2にSSH接続し、以下を実行する：

- nginx のインストール
- nginx の起動・有効化（再起動時も自動起動）
- nginx の動作確認

### Agentへの指示

以下の情報をAgentに伝えて、nginx のインストールと起動を依頼しましょう：

- EC2の接続情報（IPアドレス、SSH鍵、ユーザー名）
- インストールしたいソフトウェア（nginx）
- 起動と有効化を行うこと

<details>
<summary>📝 プロンプト例</summary>

```
EC2にSSHで接続して、nginxをインストール・起動してください。

■ 接続情報
- IP: <EC2のIPアドレス>
- SSH鍵: ~/.ssh/training-key
- ユーザー: ec2-user

■ やること
1. nginxをインストール（dnf）
2. nginxを起動
3. nginxを有効化（サーバー再起動時も自動起動するように）
4. nginxの動作確認（systemctl status）
```

</details>

### Agentの動きを観察しよう

Agentがターミナルで以下のような操作を行う様子を確認してください：

1. SSH接続コマンドを実行
2. `dnf install` でnginxをインストール
3. `systemctl start / enable` で起動・有効化
4. `systemctl status` で動作確認

> 💡 **Agent開発のポイント**: 「サーバーに入ってコマンドを実行する」という一連の作業を、自然言語の指示だけで完結できるのがAgent開発の特徴です。

`active (running)` の確認結果がAgentから報告されれば OK ✅

---

## Step 3: ブラウザでアクセスしてみよう（10分）

### やること

ブラウザで `http://<EC2のIPアドレス>` にアクセスします。

**nginxのデフォルトページ（Welcome to nginx）** が表示されれば成功 🎉

<details>
<summary>❓ ページが表示されない場合</summary>

Agentにトラブルシューティングを依頼しましょう：

```
EC2（<IPアドレス>）にSSHで接続して、以下を確認・修正してください。
- nginx が起動しているか（systemctl status nginx）
- セキュリティグループで HTTP(80) が許可されているか
```

その他のチェックポイント：
- **IPアドレスの確認**: `http://` で始まるURL（`https://`ではない）を使用しているか
- **ファイアウォール**: EC2のOS側でポート80がブロックされていないか（Amazon Linux 2023はデフォルトで許可）

</details>

## Step 4: Agent にカスタムWebページを作成・デプロイさせよう（35分）

### やること

Agentに「HTMLページを作成してEC2にデプロイして」と指示します。ファイルの作成からEC2への転送・配置まで、すべてAgentが実行します。

### ゴール

- Agentにトレーニング用のWebページ（HTML）を作成させる
- AgentがEC2に転送してnginxで公開する
- ブラウザでカスタムページが表示されることを確認する

> 💡 **ポイント**: ここでは「Agent にどう伝えればイメージ通りのページが作れるか」を考えてみましょう。デザインの要件も含めてプロンプトを工夫してください。

### Agentへの指示

以下の2つを **1つのプロンプトで** Agentに依頼してみましょう：

1. **HTMLページの作成** — どんなページにするかは自由です
   - 例: トレーニングのダッシュボードページ、自己紹介ページ、ポートフォリオなど
   - デザイン要件も含めると、よりイメージに近いものが生成されます
2. **EC2へのデプロイ** — 作成したファイルをEC2に転送して配置

<details>
<summary>📝 プロンプト例</summary>

```
以下の2つの作業を実行してください。

■ 1. HTMLページの作成
web/index.html に以下の要件でHTMLファイルを作成してください。
- タイトル: 「AI駆動IaCワークショップ ダッシュボード」
- ヘッダーに研修名と参加者名（あなたの名前）を表示
- セッション1〜6の一覧をカード形式で表示（セッション名、概要、所要時間）
- レスポンシブデザイン（スマホでも見やすい）
- CSSはHTMLファイル内にインラインで記述
- モダンなデザイン（グラデーション背景、影付きカード等）

■ 2. EC2へのデプロイ
作成した web/index.html を以下のEC2に転送して公開してください。
- IP: <EC2のIPアドレス>
- SSH鍵: ~/.ssh/training-key
- ユーザー: ec2-user
- 配置先: /usr/share/nginx/html/index.html
```

</details>

### Agentの動きを観察しよう

Agentが以下のような手順で作業する様子を確認してください：

1. HTMLファイルを作成・保存
2. `scp` でEC2にファイルを転送
3. SSHでEC2に接続してファイルを配置

> 💡 **Agent開発のポイント**: 「ファイル作成」と「サーバーへのデプロイ」という異なる種類の作業を、1つのプロンプトでまとめて指示できます。

### ブラウザで確認

`http://<EC2のIPアドレス>` にアクセスして、カスタムページが表示されることを確認 ✅

## Step 5: Agent にWebページの改善・再デプロイを指示しよう（25分）

### やること

Step 4で作成したページに機能を追加して、もう一度デプロイします。改善とデプロイの両方をAgentに指示します。

### ゴール

ページを改善して、再デプロイする一連の流れをAgent経由で体験する。

> 💡 **ポイント**: 「既存のファイルを改善してデプロイまでして」とAgentに伝えるとき、**何を変えたいのか具体的に** 伝えることが重要です。

### 改善のアイデア（好きなものを選んでください）

- 現在の時刻をリアルタイム表示するJavaScriptを追加
- ダークモード切り替えボタンを追加
- セッションの進捗を更新できるチェックボックスを追加
- アニメーション効果を追加
- AWS構成の簡易図を表示

<details>
<summary>📝 プロンプト例</summary>

```
web/index.html を改善して、EC2に再デプロイしてください。

■ 改善内容
- 現在の日時をリアルタイムで表示するセクションを追加（JavaScriptで毎秒更新）
- ダークモード/ライトモードを切り替えるトグルボタンを追加
- 各セッションにチェックボックスを追加し、完了したセッションをマークできるようにする

■ デプロイ先
- IP: <EC2のIPアドレス>
- SSH鍵: ~/.ssh/training-key
- ユーザー: ec2-user
- 配置先: /usr/share/nginx/html/index.html
```

</details>

ブラウザで `http://<EC2のIPアドレス>` をリロードして、改善が反映されていることを確認 ✅

## 📝 振り返り（5分）

### このセッションで体験したこと

| 作業 | Agent の役割 | 学び |
|------|-------------|------|
| SG にHTTPルール追加 | Terraformコード修正 + apply | 既存インフラの変更もAgentで |
| nginx インストール | SSH接続 + コマンド実行 | サーバー操作も自然言語で指示できる |
| HTML ページ作成 | ファイル生成 | デザイン含めたコード生成 |
| デプロイ | scp転送 + SSH配置 | 複数の作業を1つのプロンプトで |
| ページ改善 | 修正 + 再デプロイ | 改善→デプロイの反復もAgentで効率化 |

### プロンプトで意識したこと

- **接続情報**を正確に伝える（IP、鍵、ユーザー名、配置先）
- **複数の作業をまとめて指示**すると効率的（作成+デプロイ）
- **デザイン要件**はできるだけ具体的に伝える（色、レイアウト、機能）
- **改善依頼**は「現状の何が不満で、どうしたいか」を伝える

```
