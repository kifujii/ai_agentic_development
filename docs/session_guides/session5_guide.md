## 前半: SSH なしでサーバーを管理できるようにしよう（20分）

### チャレンジ

セキュリティチームの依頼に応えてください。**SSH を使わなくてもサーバーを管理できる仕組み** を調べて導入しましょう。

### 達成条件

- EC2 が **AWSコンソールの対象サービスの管理画面** に表示される
- **SSH を使わずに** EC2 上でコマンドを実行できる

### やること

Claude Code に「SSH を使わずにEC2を管理できる AWS のサービスを教えて。それを導入したい」と伝えてください。何のサービスを使うか、何が必要かは Claude Code と一緒に考えましょう。

> 💡 **ヒント**: EC2 上のソフトウェアが AWS のサービスと通信するには「許可」が必要です。Claude Code に相談すれば、何をどういう順番で作ればいいか教えてくれます。

<details>
<summary>🔍 困ったら: もう少し具体的なヒント</summary>

- AWS Systems Manager（SSM）というサービスが SSH の代わりになります
- EC2 が AWS サービスと通信するには **IAM ロール** が必要です
- SSM Agent のインストールは **Ansible Playbook** で行いましょう
- Amazon Linux 2023 には SSM Agent がプリインストールされている場合があります
- プレフィックスとして環境変数 `TF_VAR_prefix` の値を使ってください
- IAM の変更は反映に 1〜2 分かかることがあります

</details>

### 確認: AWSコンソールで管理画面を見てみよう

**あなたが** AWSコンソールにログインして、導入したサービスの管理画面を確認してください。

1. AWSコンソール上部の検索バーに `Systems Manager` と入力して開く
2. 左メニューから **「ノード管理」→「フリートマネージャー」** をクリック
3. EC2 インスタンスが **マネージドインスタンス** として表示されていることを確認

> 💡 表示されるまで **1〜2分** かかることがあります。表示されない場合はページをリロードしてください。

### 確認: SSH なしでコマンドを実行してみよう

導入したサービスを使って、SSH を使わずにコマンドを実行してみましょう。Claude Code に以下のように伝えてください：

```
先ほど導入したサービスの機能を使って、EC2 上で以下のコマンドを実行してください。
結果も表示してください。SSH は使わないでください。

hostname
uptime
free -m
df -h
```

> 💡 **SSH なしでサーバー管理ができる** ことを実感できましたか？運用の現場では、セキュリティグループで SSH ポートを閉じたまま管理できることが大きなメリットです。

### Ansible との比較を考えてみましょう

| 項目 | 今回導入したサービス | Ansible |
|------|---------------------|---------|
| 接続方式 | AWS API 経由 | SSH |
| 実行方法 | AWS CLI / コンソール | ターミナル |
| 適した用途 | 緊急対応、一回限りの操作 | 繰り返す定型作業、自動化 |

前半完了 ✅

## 後半: サーバーの監視基盤を構築しよう（20分）

### チャレンジ

運用チームの依頼に応えてください。EC2 の **CPU・メモリ・ディスクの使用率をリアルタイムで監視** できるようにし、**異常値でアラートが飛ぶ仕組み** も作ってください。

### 達成条件

- AWS の監視サービスに EC2 のカスタムメトリクス（CPU/メモリ/ディスク）が表示される
- システムログが収集されている
- CPU 使用率が高くなったら **アラーム** で通知される

### やること

Claude Code に「EC2 の CPU やメモリの使用率をリアルタイムで監視したい。異常時にアラートも欲しい」と伝えてください。どの AWS サービスを使うかも含めて相談しましょう。

> 💡 **ヒント**: 前半と同じように「インストール → 設定 → 起動」の流れです。アラームは AWS CLI で作成できます。

<details>
<summary>🔍 困ったら: もう少し具体的なヒント</summary>

- CloudWatch Agent というソフトウェアでメトリクスとログを収集できます
- インストール・設定は **Ansible Playbook** で行いましょう
- 設定ファイルでは収集するメトリクス（CPU、メモリ、ディスク）とログを指定します
- プレフィックスとして環境変数 `TF_VAR_prefix` の値を使ってください
- CloudWatch Alarm は **AWS CLI** で作成すると簡単です
- メトリクスが表示されるまで **2〜5分** かかります

</details>

### 確認: AWSコンソール + CLI で結果を確認しよう（10分）

Agent の起動後、**数分待ってから** 以下を確認します。

**メトリクスの確認（AWSコンソール）**:

1. **あなたが** AWSコンソール → CloudWatch → メトリクス → すべてのメトリクス を開く
2. **カスタム名前空間** の一覧から `<PREFIX>/EC2` を探す（例: `user01/EC2`）
3. メトリクス（CPU、メモリ、ディスク）が表示されていることを確認

**ロググループの確認（AWS CLI）**:

```bash
aws logs describe-log-groups --log-group-name-prefix "/${TF_VAR_prefix}/ec2" --query "logGroups[].logGroupName" --output table
```

**アラームの確認（AWSコンソール）**:

1. AWSコンソール → CloudWatch → アラーム を開く
2. `<PREFIX>` を含むアラームが表示されていることを確認

> 💡 現時点ではCPU使用率が低いため、アラームのステータスは「OK」のはずです。

メトリクス・ロググループ・アラームが確認できれば後半完了 ✅
