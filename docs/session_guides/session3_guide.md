# セッション3：Web サイトを HTTPS 対応にしよう（任意・45分）

> このセッションは **任意（発展課題）** です。セッション2が完了し、余裕がある方向けです。

## 🎯 このセッションの到達状態

Webサイトが HTTPS で配信され、`http://` へのアクセスが自動的に `https://` にリダイレクトされる状態になっています。

![目標構成](../images/session3_target.svg)

### セッション2からの変化

| | セッション2 | セッション3 |
|---|:---:|:---:|
| プロトコル | HTTP (80) | **HTTPS (443)** |
| 証明書 | なし | 自己署名証明書 |
| セキュリティ | 通信が暗号化されない | **通信が暗号化される** |

> 🎓 本番環境では Let's Encrypt や ACM などの正規の証明書を使いますが、このセッションでは学習のため **自己署名証明書** を使います。

## Step 1: セキュリティグループに HTTPS(443) を追加しよう（10分）

### やること

HTTPS 通信を許可するため、セキュリティグループに **443番ポート** を追加します。

### ゴール

- セキュリティグループに **HTTPS(443)** のインバウンドルールが追加されている（ソース: `0.0.0.0/0`）
- 既存の SSH(22) と HTTP(80) のルールがそのまま残っている
- `terraform apply` が成功している

<details>
<summary>📝 プロンプト例</summary>

```
terraform/vpc-ec2/ の既存コードで、EC2のセキュリティグループに HTTPS(443) のインバウンドルールを追加してください。
ソースは 0.0.0.0/0 で、既存の SSH(22) と HTTP(80) のルールはそのまま残してください。

terraform apply まで実行してください。
```

</details>

### 確認

```bash
terraform -chdir=terraform/vpc-ec2 output
```

セキュリティグループにポート443のルールが追加されていれば OK ✅

## Step 2: SSL証明書の作成と nginx の HTTPS 設定（20分）

### やること

EC2上で自己署名SSL証明書を作成し、nginx を HTTPS に対応させます。

### ゴール

1. EC2上に **自己署名SSL証明書** が作成されている
2. **nginx** が HTTPS(443) でリクエストを受け付けている
3. **HTTP(80)** へのアクセスが **HTTPS に自動リダイレクト** される

### Agentへの指示

以下の要件を伝えて、SSL対応を実施してもらいましょう：

- EC2の接続情報
- 自己署名証明書を `openssl` で作成すること
- nginx にSSL設定を追加すること
- HTTP → HTTPS のリダイレクトも設定すること

<details>
<summary>📝 プロンプト例</summary>

```
EC2にSSHで接続して、以下の作業を実行してください。

■ 接続情報
- IP: <EC2のIPアドレス>
- SSH鍵: keys/training-key
- ユーザー: ec2-user

■ 1. 自己署名SSL証明書の作成
- openssl で秘密鍵と証明書を作成
- 保存先: /etc/nginx/ssl/（ディレクトリがなければ作成）
- 有効期間: 365日
- Common Name (CN): localhost

■ 2. nginx の HTTPS 設定
- ポート443でSSLを有効化し、作成した証明書を使用
- セッション2のHTMLコンテンツをそのまま配信

■ 3. HTTP → HTTPS リダイレクト
- ポート80へのアクセスをポート443にリダイレクト（301）

■ 4. nginx を再起動して反映
```

</details>

## Step 3: ブラウザで確認しよう（10分）

### HTTP → HTTPS リダイレクト確認

ブラウザで `http://<EC2のIPアドレス>` にアクセスします。

自動的に `https://<EC2のIPアドレス>` にリダイレクトされれば OK ✅

### HTTPS アクセス確認

ブラウザに **「この接続ではプライバシーが保護されません」** という警告が表示されます。

> 💡 これは自己署名証明書を使っているため正常な動作です。本番では正規の証明書を使えば、この警告は出ません。

「詳細設定」→「<IPアドレス>にアクセスする（安全ではありません）」をクリックして進みます。

- ✅ セッション2で作成したWebページが表示される
- ✅ URLが `https://` で始まっている
- ✅ ブラウザのアドレスバーに鍵アイコン（⚠️付き）が表示される

上記が確認できれば **セッション3完了** 🎉

<details>
<summary>❓ ページが表示されない場合</summary>

Agentにトラブルシューティングを依頼しましょう：

- **症状**: ブラウザに何が表示されるか（接続拒否、タイムアウト、など）
- **確認してほしいこと**:
  - nginx の設定テスト（`nginx -t`）
  - nginx のエラーログ（`/var/log/nginx/error.log`）
  - セキュリティグループでポート443が許可されているか
  - 証明書ファイルが存在するか

</details>

## 📝 振り返り（5分）

### HTTPS 対応で学んだこと

| 要素 | 内容 |
|------|------|
| **SSL証明書** | サーバーの身元を証明し、通信を暗号化するファイル |
| **自己署名証明書** | 学習・開発用。本番では CA 発行の証明書を使う |
| **nginx SSL設定** | `ssl_certificate` / `ssl_certificate_key` で証明書を指定 |
| **HTTPリダイレクト** | `return 301 https://...` で HTTP → HTTPS に転送 |
| **セキュリティグループ** | ポート443を開けないと HTTPS 通信が届かない |

### インフラ変更 + サーバー設定の組み合わせ

このセッションでは **Terraform**（SG変更）と **サーバー設定**（証明書・nginx）の両方を変更しました。実務でも、ネットワーク側（ファイアウォール/SG）とサーバー側（アプリ設定）の**両方を正しく変更する**必要がある場面は多くあります。

## ファイル構成

```
terraform/
└── vpc-ec2/
    ├── main.tf          # SG に HTTPS(443) ルールが追加された
    ├── variables.tf
    └── outputs.tf

# EC2上のファイル
/etc/nginx/ssl/
├── server.key           # SSL秘密鍵
└── server.crt           # SSL証明書
/etc/nginx/              # nginx 設定（HTTPS + リダイレクト）
```

## ⚠️ リソースの削除

このセッションで追加・変更したリソースは以下のとおりです：

| リソース | 削除方法 |
|---------|---------|
| セキュリティグループの443ルール | `terraform -chdir=terraform/vpc-ec2 destroy` で VPC/EC2 ごと削除 |
| EC2上のSSL証明書・nginx設定 | EC2 削除時に一緒に消える |

> 💡 セッション3単独で追加したAWSリソースはセキュリティグループのルールのみです。EC2/VPC の削除はワークショップ終了時にまとめて実施します（[セッション6](session6_guide.md) のクリーンアップ手順を参照）。
