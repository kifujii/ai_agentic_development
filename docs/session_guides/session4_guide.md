# セッション4：Ansible によるサーバー運用自動化（必須・1時間）

## 学習目標

- Terraform と Ansible の役割の違いを学ぶ
- AI Agent を用いたトラブルシューティングを体験する
- 障害対応を Playbook として自動化する運用改善の流れを体験する

## 🔰 Ansible とは？

ここまでのセッションでは **Terraform** を使って AWS のインフラ（VPC、EC2 など）を構築してきました。このセッションからは **Ansible** という別のツールを使います。

| | Terraform | Ansible |
|---|-----------|---------|
| **役割** | インフラを**作る** | 作ったサーバーの**中身を整える** |
| **対象** | VPC、EC2、セキュリティグループなど | パッケージ、設定ファイル、サービスなど |
| **接続方法** | AWS API | SSH |
| **設定ファイル** | `.tf`（HCL） | Playbook（YAML） |

> 💡 Terraform が「家を建てる」ツールなら、Ansible は「家具を配置して住めるようにする」ツールです。どちらも **コードで管理** する点は共通しており、一緒に使うのが一般的です。

Ansible の特徴:
- **エージェントレス**: サーバーに専用ソフトを入れなくていい（SSH で接続するだけ）
- **Playbook**: やりたいことを YAML で書く（実行手順書のようなもの）
- **冪等性（べきとうせい）**: 何回実行しても同じ結果になる（2回目は「変更なし」になる）

</details>

## Step 2: 接続テスト（5分）

### やること

Ansible の `ping` モジュールで EC2 への接続を確認します。

Claude Code に `プロジェクトルートから Ansible の接続テスト（ansible all -m ping）を実行して` と指示しましょう。

`web1 | SUCCESS` と表示されれば OK ✅

> 💡 手動で実行する場合（プロジェクトルートから）：
> ```bash
> ANSIBLE_CONFIG=ansible/ansible.cfg ansible -i ansible/inventory.ini all -m ping
> ```

<details>
<summary>❓ 接続できない場合</summary>

接続に失敗した場合は、**エラーメッセージをそのまま Claude Code に共有** してみましょう。AI Agent がエラーの原因を分析し、修正してくれます。

よくある原因：
- `inventory.ini` のIPアドレスが正しいか確認（プレースホルダー `<EC2のIP>` のままになっていませんか？）
- EC2が起動しているか確認
- IPアドレスが正しいか確認（`terraform -chdir=terraform/vpc-ec2 output instance_public_ip`）
- SSH鍵の権限を確認（`chmod 400 keys/training-key`）
- セキュリティグループでSSHが許可されているか確認

</details>

> 💡 **トラブルシューティングの基本パターン**: セッション1で紹介した「エラー → Claude Code に共有 → 原因分析 → 修正 → 再確認」のパターンをここでも活用しましょう。何か問題が起きたら、エラーメッセージをそのまま Claude Code に貼り付けてください。

## Step 6: 🔧 障害対応シミュレーション — EC2に障害発生！（15分）

### やること

実務では、サーバーに予期しない障害が発生します。このステップでは **スクリプトで EC2 にランダムな障害を発生** させ、Claude Code と協力して **原因調査 → 診断 → 復旧** を体験します。

> 🎓 **このシミュレーションの特徴**: 障害の内容はスクリプトがランダムに選択するため、**あなたも Claude Code もどんな障害が起きたか事前にはわかりません**。実務と同じ「何が起きたかわからない」状態から調査を始める体験ができます。

---

### Phase 1: 障害を発生させる — あなたがスクリプトを実行（2分）

**あなたのターミナルで** 以下のコマンドを実行してください。スクリプトが EC2 にランダムな障害を発生させます：

```bash
./scripts/break_session.sh <EC2のIP>
```

> ⚠️ `<EC2のIP>` は事前準備でメモした実際のIPアドレスに置き換えてください。

スクリプトが完了すると、Claude Code に伝えるべきメッセージが表示されます。

> ⚠️ **重要**: スクリプトの中身を確認しないでください。障害の内容を知ってしまうと、シミュレーションの効果がなくなります。

---

### Phase 2: Claude Code にトラブルシューティングを依頼する（8分）

スクリプトが表示したメッセージを参考に、**Claude Code に** 障害の調査と復旧を依頼します。

<details>
<summary>📝 プロンプト例</summary>

```
EC2（<EC2のIP>）に何らかの問題が発生しています。
SSHで接続して、サーバーの状態を調査し、
見つかった問題をすべて修正してください。

■ 接続情報
- IP: <EC2のIP>
- SSH鍵: keys/training-key
- ユーザー: ec2-user

■ 調査ポイント
- サービスの状態（systemctl で確認）
- 設定ファイルの整合性
- ログファイル
- ディスク使用量
- 不審なプロセスやジョブ
```

</details>

### Claude Code が行うこと（観察してください）

障害の内容によって調査フローは変わりますが、Claude Code は概ね以下のようなプロセスで問題を解決するはずです：

1. **SSH で EC2 に接続** し、全体的な状況を確認
2. **サービスの状態確認**: `systemctl` で主要サービスをチェック
3. **ログの確認**: `/var/log/messages` などを調査
4. **設定ファイルの確認**: 不正な設定がないかチェック
5. **ディスク・リソースの確認**: `df`, `free` などでリソース状況を確認
6. **問題の特定と修復**: 発見した問題を1つずつ修復
7. **修復後の確認**: サービス再起動、動作テスト

> 💡 **ここがポイント**: Claude Code は「症状の確認 → 仮説の立案 → 調査 → 原因特定 → 修復 → 確認」と、**複数のステップを自動的に繰り返して** 問題を解決します。人間が1つずつコマンドを打って調べるのと同じ思考プロセスを AI が実行しています。

Claude Code がすべての問題を特定・修復したら **復旧成功** ✅

> 💡 復旧後は `http://<EC2のIP>` にブラウザでアクセスして、nginx のページが正常に表示されることもあわせて確認しましょう。

**エンドユーザー視点での確認**も行いましょう：

- ブラウザで `http://<EC2のIPアドレス>` にアクセスして、nginx のページが表示されることを確認

> 💡 サービスレベルでの復旧だけでなく、**実際にユーザーがアクセスできるか**を確認するのは、実務のインシデント対応でも重要なステップです。

> 💡 **もし Claude Code が一部の問題を見落としたら？**: 「まだ他にも問題があるかもしれません。もう一度全体をチェックしてください」と追加で指示してみましょう。

## 📝 振り返り（5分）

### TerraformとAnsibleの使い分け

| Terraform（セッション1〜3） | Ansible（セッション4〜6） |
|:---:|:---:|
| リソースの **作成・変更・削除** | サーバーの **設定・運用** |
| AWSリソースを構築・管理 | 構築済みサーバーを操作 |
| `terraform apply` / `destroy` | `ansible-playbook` |

### このセッションで作ったPlaybook

| Playbook | 用途 | 実務での活用場面 |
|----------|------|-----------------|
| 状態確認 | サーバーの健康状態を把握 | 定期的なヘルスチェック |
| 再起動 | 安全な再起動と前後確認 | メンテナンス作業 |
| サービス管理 | サービスの起動/停止 | 障害対応・デプロイ |
| 診断・復旧 | 問題の発見と対処 | **障害対応（インシデントレスポンス）** |

### 🔧 トラブルシューティングの学び

| 体験 | 学んだこと |
|------|-----------|
| ランダムな障害が発生し、AI Agent が調査・復旧 | AI Agent は「症状確認 → 仮説 → 調査 → 原因特定 → 修復 → 確認」を自動で繰り返す |
| 何が起きたかわからない状態から調査開始 | **実務と同じ「予測不能な障害」に対する対応力が身につく** |
| AI Agent の調査・復旧 → Playbook 化 | **一度対応した作業を次回のために自動化する** |

> 🎓 **このパターンを覚えておきましょう**: 何か問題が起きたら → **あなたが状況を Claude Code に伝える** → Claude Code が調査・原因特定・修復。このパターンはセッション5以降でも使えます。

### 📖 コードを理解しよう — Playbook の構造を読み解く

Ansible の Playbook は YAML で書かれた「手順書」です。各 Playbook の **処理の流れと各タスクの役割** を人に説明できるようになりましょう：

<details>
<summary>📝 プロンプト例</summary>

```
ansible/playbooks/ にある全 Playbook について、以下の内容を含む解説ドキュメントを作成してください。
保存先: docs/session4_design.md

■ 含めてほしい内容
1. 各 Playbook の処理フロー図（テキストベース）
2. Playbook の基本構造の解説（hosts, become, vars, tasks の役割）
3. 使用しているモジュール一覧と各モジュールの機能
   （例: systemd → サービス管理、command → コマンド実行、debug → 結果表示）
4. register と changed_when の仕組みと使い分け
5. ignore_errors の意味と使いどころ
6. server_health_check.yml の条件分岐（when句）がどう動くかの詳細解説
```

</details>

生成されたドキュメントを読んで、以下を確認しましょう：

- [ ] Playbook の `hosts`, `become`, `tasks` が何を意味するか説明できる
- [ ] `register` で取得した値を `debug` で表示する仕組みが説明できる
- [ ] `systemd` モジュールと `command` モジュールの使い分けが説明できる
- [ ] server_health_check.yml の条件分岐（`when`）がどう動くか説明できる

## ファイル構成

```
ansible/
├── inventory.ini          # 接続先定義
├── ansible.cfg            # 基本設定
└── playbooks/
    ├── (状態確認 Playbook)
    ├── (再起動 Playbook)
    ├── (サービス管理 Playbook)
    └── (診断・復旧 Playbook)
```

> 💡 Playbook のファイル名やタスクの構成は Claude Code に任せて大丈夫です。受講者ごとに異なる構成になることが期待されます。ただし、完了チェック（`check.sh`）はファイル名のパターンで検出するため、Playbook 名に `check`、`restart`、`service`、`health` などの用途がわかる名前を含めると安心です。
